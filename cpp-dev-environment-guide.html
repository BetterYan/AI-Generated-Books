Windows 跨平台 C++ 开发环境搭建指南

从零开始，在 Windows 上搭建一套完整的跨平台 C++ 开发环境。本指南会详细解释每个工具的作用、安装步骤和验证方法，并穿插一些行业趣闻，让你对工具有更深的理解。

总览

整体架构概览

在 Windows 上做跨平台 C++ 开发，你需要理解一个核心问题：Windows 和 Linux/macOS 使用不同的编译器、不同的 C 运行时库、不同的系统 API。你的代码需要在所有这些平台上编译通过并正确运行。因此，开发环境的核心思路是：

在 Windows 上用 MSVC 或 Clang 做日常开发和调试，同时通过 WSL2 或 MinGW 验证代码在 Linux/GCC 环境下的行为。使用 CMake 作为统一的构建系统，确保同一份构建脚本在所有平台上都能工作。

行业故事：为什么 C++ 跨平台这么"痛"？

故事要从 1983 年说起。Bjarne Stroustrup 在贝尔实验室创造 C++ 时，C 语言已经以"可移植"著称——但 C 的可移植性其实是个美丽的谎言。不同平台的 int 大小不同、字节序不同、系统调用完全不同。C++ 继承了这个问题，还引入了更多复杂性：名字修饰（name mangling）没有统一标准，异常处理的 ABI 各异，甚至 #include 的文件名大小写敏感性在 Windows 和 Linux 上都不同。

这就是为什么在 2000 年代，许多 C++ 团队不得不维护完全独立的项目文件（.vcproj / Makefile / .xcodeproj）。直到 CMake 在 2000 年由 Bill Hoffman 创建（灵感来自 KDE 项目的构建需求），C++ 社区才有了真正的跨平台构建标准。如今 CMake 已成为事实标准，被 Google、NASA、CERN 等机构广泛使用。

编译器

MSVC 编译器

基础知识

MSVC（Microsoft Visual C++）是微软自家的 C++ 编译器，包含在 Visual Studio 中。它是 Windows 平台上最成熟、兼容性最好的 C++ 编译器，对 Windows SDK 和 Win32 API 的支持无人能及。

MSVC 的主要特点是：编译速度较快，错误提示清晰（尤其是 VS2022 之后的改进），对 C++ 最新标准的支持积极（C++20 modules、C++23 等），并且与 Windows 调试器完美集成。

行业故事：MSVC 的"C++ 方言"时代

在 2010 年之前，MSVC 曾因为对 C++ 标准支持不佳而饱受诟病。许多在 GCC 上能编译的标准 C++ 代码在 MSVC 上会报错。这导致了一个有趣的现象：开源社区经常把"MSVC 兼容"当作一个专门的标签。著名游戏引擎 Unreal Engine 的代码风格——大量使用宏、自定义智能指针而非标准库——部分原因就是早期要兼容各种编译器的怪异行为。

转机出现在 2015 年。微软在 Visual Studio 2015 中推出了重大更新的 MSVC 工具集（v140），开始积极追赶 C++ 标准。到 VS2019/VS2022，MSVC 已经成为一个优秀的标准兼容编译器，甚至在某些 C++20/23 特性上领先 GCC 和 Clang。

GCC (MinGW-w64)

基础知识

GCC（GNU Compiler Collection）是 Linux 世界的标准编译器。MinGW-w64 是 GCC 在 Windows 上的移植版本，能让你在 Windows 上用 GCC 编译 C++ 程序，并且链接 Windows 的系统库（而非 Cygwin 的 POSIX 层）。

为什么 Windows 上还需要 GCC？因为你的代码最终可能要在 Linux 服务器上运行，而 GCC 和 MSVC 对 C++ 标准的解读有细微差异。在 Windows 上用 MinGW 编译一遍，能提前发现一些跨编译器兼容性问题。

行业故事：MinGW 与 Cygwin 的路线之争

2000 年代初期，在 Windows 上使用 GCC 有两条路线：Cygwin（通过 POSIX 兼容层模拟 Linux 环境）和 MinGW（直接调用 Windows 原生 API）。Cygwin 的优势是兼容性好——大量 Linux 代码几乎不用改就能编译。代价是性能损失和运行时依赖 cygwin1.dll。

MinGW 选择了更难但更纯粹的路：只生成原生 Windows 可执行文件，不需要额外的 DLL。这导致很多 Linux 代码编译不过（因为没有 fork()、unistd.h 等），但产出的程序性能和原生程序一样。游戏引擎、嵌入式工具等对性能敏感的项目偏爱 MinGW。

如今，MinGW-w64（MinGW 的 64 位分支）已经成为主流，被 MSYS2 项目维护和分发。而 Cygwin 则更多用于学术研究和工具移植场景。

安装步骤

推荐通过 MSYS2 安装 MinGW-w64，这是最方便的方式：

2

打开 MSYS2 终端，更新包数据库

Clang / LLVM

基础知识

Clang 是 LLVM 项目中的 C/C++ 编译器前端。它以编译错误信息极其友好著称——当你的代码有语法错误时，Clang 不仅告诉你错在哪一行，还会用彩色箭头标出确切位置，甚至给出修复建议。

Clang 的另一个杀手锏是工具链生态：clang-format（代码格式化）、clang-tidy（静态分析）、clangd（语言服务器，为编辑器提供智能补全）。这些工具在现代 C++ 开发中几乎是标配。

行业故事：Clang 如何改变了编译器竞争格局

2007 年，Apple 对 GCC 的进展不满——GCC 的代码库臃肿、编译速度慢、错误信息晦涩，而且 GCC 的许可证（GPL）限制了 Apple 将其深度集成到 Xcode 中。于是 Apple 大力投资 Chris Lattner 在伊利诺伊大学发起的 LLVM/Clang 项目。

Clang 从设计之初就追求三件事：极致的编译速度、清晰的错误信息、模块化的架构（方便第三方工具复用编译器前端）。2009 年，Clang 首次完整编译了 FreeBSD 内核；2013 年，Google 开始在 Chrome 项目中使用 Clang。

最有意思的是微软的反应。面对 Clang 的崛起，微软没有固守 MSVC，而是在 VS2019 中推出了 "Clang/LLVM for Windows"（也叫 clang-cl），让开发者可以在 Visual Studio 中使用 Clang 作为后端编译器。这标志着 Clang 已经从一个"Apple 的项目"变成了整个行业的标准。

安装步骤

Clang 已包含在 Visual Studio 2022 中，也可单独安装：

1

方式一：通过 Visual Studio Installer

在"单个组件"中搜索并勾选 C++ Clang for Windows 和 C++ Clang-cl for v143。

2

方式二：独立安装（推荐，获得完整工具链）

前往 LLVM GitHub Releases 下载最新的 LLVM-*-win64.exe 安装包。安装时勾选 "Add LLVM to the system PATH"。

验证安装

# 预期输出：clang version 18.x.x

# 预期输出：clang-format version 18.x.x

# 预期输出：LLVM version 18.x.x

# 编译测试

.\test.exe
# 预期输出：Clang OK
构建系统

CMake

基础知识

CMake 是 C++ 世界的"构建系统的构建系统"。它本身不编译代码，而是根据你的 CMakeLists.txt 配置文件，生成对应平台的原生构建文件——在 Windows 上生成 VS 解决方案或 Ninja 文件，在 Linux 上生成 Makefile，在 macOS 上生成 Xcode 项目。

这意味着你只需要维护一份 CMakeLists.txt，就能在所有平台上构建项目。这是 C++ 跨平台开发的基石。

行业故事：CMake 的诞生——KDE 项目的"救星"

2000 年，KDE 桌面环境的开发者们面临一个痛苦的问题：他们使用的 autoconf/automake 构建系统极其复杂——一个 configure.in 文件动辄上千行，M4 宏语法晦涩难懂，而且在 Windows 上几乎不可用。

Kitware 公司的 Bill Hoffman 当时正在为美国国家医学图书馆开发可视化软件 VTK。他也饱受构建系统之苦，于是创建了 CMake（Cross-platform Make），核心理念是"简单、跨平台、不依赖特定 IDE"。KDE 在 2006 年从 KDE 3 到 KDE 4 的迁移中全面采用了 CMake，这一决定极大地推动了 CMake 的普及。

如今 CMake 已经成为 C++ 的事实标准构建系统。Google、Facebook、NASA JPL（用于火星探测器的飞行软件）、CERN（大型强子对撞机的控制软件）都在使用它。2019 年 CMake 3.15 引入了对 Windows 上 MSVC 的一流支持后，更是巩固了不可动摇的地位。

安装步骤

1

前往 cmake.org/download 下载 Windows x64 Installer（cmake-*-windows-x86_64.msi）。

安装时务必勾选 "Add CMake to the system PATH"（选择 "for all users" 或 "for current user" 均可）。

注意

Visual Studio 2022 也内置了一份 CMake，但版本可能不是最新的。建议单独安装最新版 CMake 并优先使用它。

验证安装

PowerShell
cmake --version
# 预期输出：cmake version 3.28.x 或更高
#            CMake suite maintained and supported by Kitware

# 生成测试项目
mkdir cmake-test; cd cmake-test

# 创建最简单的 CMakeLists.txt
echo 'cmake_minimum_required(VERSION 3.20)' > CMakeLists.txt
echo 'project(HelloTest LANGUAGES CXX)' >> CMakeLists.txt
echo 'set(CMAKE_CXX_STANDARD 20)' >> CMakeLists.txt
echo 'add_executable(hello main.cpp)' >> CMakeLists.txt

# 创建源文件
echo '#include <iostream>' > main.cpp
echo 'int main() { std::cout << "CMake OK" << std::endl; }' >> main.cpp

# 生成并构建
cmake -B build -G Ninja
cmake --build build
.\build\hello.exe
# 预期输出：CMake OK

Ninja

基础知识

Ninja 是一个专注于速度的小型构建工具，由 Google 的 Evan Martin 为 Chromium 项目开发。它的设计哲学是：只负责"构建"这一步，做到极致快，而把依赖分析和项目配置的复杂工作交给 CMake 等上层工具。

与传统的 nmake（Windows）或 make（Linux）相比，Ninja 的构建速度通常快 2-5 倍，在大型项目中差距更明显。在 Visual Studio 中使用 CMake 时，Ninja 是默认的构建后端。

行业故事：为什么 Chromium 需要 Ninja？

Chromium（Chrome 浏览器的开源版本）是人类历史上最大的 C++ 项目之一——超过 3500 万行代码，编译一次需要处理数万个源文件。2010 年之前，Chromium 使用 make 和 MSBuild 构建，但速度极慢：增量编译一次也要好几分钟，全量编译需要数小时。

Google 工程师 Evan Martin 分析了原因：传统构建工具在启动时就要花大量时间解析 Makefile，而 Chromium 的 Makefile 经过层层展开后极其庞大。于是他创建了 Ninja——一个极简的构建执行器。Ninja 的构建文件（build.ninja）是机器生成的、不可读的，但解析速度极快。

结果立竿见影：Chromium 的增量编译时间从几分钟缩短到几秒。Ninja 很快被 Android、LLVM 等大型项目采用，如今已成为 C++ 开发的标准配置。

IDE / 编辑器

Visual Studio 2022

基础知识

Visual Studio 是 Windows 平台上最强大的 C++ IDE。对于从 C# 转过来的开发者来说，VS 应该是非常熟悉的环境。VS2022 对 C++ 的支持做了大幅改进：原生的 CMake 项目支持、IntelliSense 智能补全、集成的调试器、性能分析器等。

配置 CMake 项目

1

用 VS2022 打开一个包含 CMakeLists.txt 的文件夹（File → Open → Folder）。

2

VS 会自动识别 CMake 项目，并创建 CMakeSettings.json。建议配置如下：

CMakeSettings.json
{
  "configurations": [
    {
      "name": "x64-Debug",
      "generator": "Ninja",
      "configurationType": "Debug",
      "buildRoot": "${projectDir}\\out\\build\\${name}",
      "installRoot": "${projectDir}\\out\\install\\${name}",
      "cmakeCommandArgs": "",
      "buildCommandArgs": "-v",
      "ctestCommandArgs": "",
      "inheritEnvironments": ["msvc_x64_x64"],
      "variables": [
        { "name": "CMAKE_TOOLCHAIN_FILE",
          "value": "C:/vcpkg/scripts/buildsystems/vcpkg.cmake" }
      ]
    },
    {
      "name": "x64-Release",
      "generator": "Ninja",
      "configurationType": "Release",
      "inheritEnvironments": ["msvc_x64_x64"]
    }
  ]
}
提示

CMAKE_TOOLCHAIN_FILE 指向 vcpkg 的 toolchain 文件后，VS 会自动使用 vcpkg 管理的第三方库。后面安装 vcpkg 时会详细说明。

行业故事：VS Code 如何统一了编辑器体验

在 VS Code 出现之前（2015 年），C++ 开发者在不同平台上使用完全不同的编辑器：Windows 上用 Visual Studio，Linux 上用 Vim/Emacs/KDevelop，macOS 上用 Xcode。每个编辑器的快捷键、配置方式、插件生态完全不同。

VS Code 的出现改变了一切。它基于 Electron（Web 技术栈），在所有平台上提供完全一致的体验。更关键的是 Language Server Protocol（LSP）——由 VS Code 团队在 2016 年提出的协议，让任何语言服务器（如 clangd）都能为任何编辑器提供智能补全、跳转定义、重构等功能。

今天，无论你用 VS Code、Neovim 还是 Emacs，都可以接入 clangd 获得相同的 C++ 分析能力。这种"编辑器无关"的工具链生态，是 C++ 社区近年来最大的进步之一。

CLion（可选）

基础知识

CLion 是 JetBrains（IntelliJ、Rider 的开发商）推出的 C++ IDE。如果你是 Rider 或 IntelliJ 用户，CLion 的界面和快捷键会让你感到非常亲切。CLion 对 CMake 的支持是一等公民级别的——打开一个 CMakeLists.txt 就能自动识别项目结构。

CLion 是付费软件（有免费试用），适合预算充足且喜欢 JetBrains 生态的开发者。

行业故事：JetBrains 为什么做 CLion

JetBrains 最初以 Java IDE IntelliJ IDEA 闻名。2010 年左右，他们开始探索 C++ 领域。团队发现：C++ 开发者的工具体验远不如 Java/C# 开发者——没有好用的重构工具、没有智能的代码分析、调试体验粗糙。

CLion 于 2015 年发布，核心卖点是将 IntelliJ 级别的代码分析带入 C++ 世界。CLion 自研了 C++ 解析引擎（而非依赖 Clang），能理解复杂的模板代码和宏展开，提供精准的重构建议。不过后来 CLion 也集成了 Clangd，实现了"自研 + Clangd"双引擎架构。

有趣的是，JetBrains 自己的 C++ 项目（如 Kotlin/Native 编译器）就是用 CLion 开发的——真正的"dogfooding"。

Git

基础知识

作为 C# 开发者，你肯定已经熟悉 Git。在 C++ 跨平台开发中，Git 有一些额外的注意事项：

  • 行尾符：Windows 使用 CRLF（\r\n），Linux/macOS 使用 LF（\n）。跨平台项目必须统一为 LF。

  • .gitignore：C++ 项目的构建产物（build/、out/、*.obj、*.exe）必须忽略。

行业故事：CRLF 引发的"百万行 diff"惨案

2012 年，一个大型开源 C++ 项目的新贡献者（使用 Windows）提交了一个 pull request。代码审查者打开 diff 一看——几乎所有文件都被标记为"已修改"，diff 高达数十万行。仔细检查后发现：这位贡献者的 Git 配置将 LF 自动转换成了 CRLF，导致每个文件的每一行都发生了变化。

这次事件后，项目维护者在 .gitattributes 文件中加入了强制行尾符配置。这也是为什么我们推荐使用 core.autocrlf false 并让 .gitattributes 来统一管理的做法。以下是推荐的 .gitattributes 配置：

vcpkg 包管理器

基础知识

vcpkg 是微软开发的 C++ 包管理器，类似于 C# 的 NuGet。它从源代码下载、编译并安装第三方 C++ 库（如 Boost、fmt、SDL2、OpenCV 等），并通过 CMake 的 toolchain 文件自动集成到你的项目中。

C++ 的包管理比 C# 复杂得多——因为 C++ 库需要针对你的编译器、C++ 标准版本、构建配置（Debug/Release）分别编译。vcpkg 替你处理了这些复杂工作。

行业故事：C++ 为什么需要包管理器

在 NuGet、npm、pip 等包管理器诞生多年后，C++ 社区仍然在"手动下载源码、编译、复制头文件、配置链接路径"的痛苦循环中。2016 年之前，一个典型的 C++ 项目引入 Boost 库的步骤是：下载源码 → 安装 b2 构建工具 → 花 30 分钟编译 → 手动设置 INCLUDE 和 LIB 路径 → 祈祷版本兼容。

2016 年，微软的 Steve MacRobert 在 CppCon 大会上发布了 vcpkg。核心理念是：从源码编译（保证二进制兼容）、与 CMake 无缝集成、跨平台统一体验。到 2024 年，vcpkg 已经包含超过 2000 个库的"port"（构建配方），成为 C++ 包管理的事实标准之一（另一个竞争者是 Conan）。

一个有趣的事实：vcpkg 自身也是用 C++ 写的，所以它也是自己的用户——开发团队称之为 "self-hosting"。

WSL2 子系统

基础知识

WSL2（Windows Subsystem for Linux 2）让你在 Windows 上运行真正的 Linux 内核。对于跨平台 C++ 开发，WSL2 的价值在于：你可以在同一台机器上用 Linux GCC 编译和测试代码，确保它在 Linux 服务器上的行为正确。

WSL2 相比 WSL1 有本质提升：运行完整的 Linux 内核、完全的系统调用兼容性、更好的文件 I/O 性能（在 Linux 文件系统内）。

行业故事：微软拥抱 Linux 的里程碑

2016 年，当微软宣布在 Windows 10 中引入 WSL（当时叫 "Bash on Ubuntu on Windows"）时，整个科技界震惊了。要知道，微软前 CEO Steve Ballmer 曾在 2001 年公开称 Linux 为"癌症"。

WSL 的诞生背后有一个有趣的技术合作：微软与 Canonical（Ubuntu 母公司）的工程师们花了大量时间解决 Windows 内核和 Linux 系统调用之间的翻译问题。最初的 WSL1 是一个"翻译层"，将 Linux 系统调用翻译成 Windows NT 内核调用——这个工程量之大，堪比当年 Wine（在 Linux 上运行 Windows 程序）的难度。

2019 年推出的 WSL2 采用了更直接的方式：在 Hyper-V 轻量虚拟机中运行真正的 Linux 内核。这一决定让兼容性从"95%"提升到了"99.9%"，也让 C++ 开发者可以放心地在 WSL2 中使用 GCC、Valgrind、GDB 等 Linux 工具链。

# 设置用户名和密码

行业故事：调试器如何改变了软件行业

1947 年 9 月 9 日，哈佛大学 Mark II 计算机出故障了。Grace Hopper（后来成为 COBOL 语言之母）的同事们在继电器中发现了一只飞蛾。他们把飞蛾贴在日志本上，写道："First actual case of bug being found."——这就是"debug"一词的由来。

在 C++ 的世界中，调试比 C# 更具挑战性。C# 的 GC 确保不会出现悬垂指针，异常系统确保错误被正确传播。而 C++ 的未定义行为（UB）可能让程序在完全无关的地方崩溃——一个数组越界写入可能在几百万次操作后才触发段错误。这就是为什么 C++ 社区如此重视 Sanitizer 工具（下一章会讲到），它们能在运行时检测大部分 UB，从根本上改变了 C++ 的调试方式。

Sanitizer 工具

基础知识

Sanitizer 是一类在编译时注入检测代码的运行时分析工具，能在程序运行过程中检测各种内存错误和未定义行为。它们是现代 C++ 开发中最重要的质量保证工具。

Sanitizer检测什么编译器支持性能开销
AddressSanitizer (ASan)缓冲区溢出、use-after-free、内存泄漏GCC, Clang, MSVC~2x 减速
UndefinedBehaviorSanitizer整数溢出、空指针解引用、对齐错误GCC, Clang~15% 减速
ThreadSanitizer (TSan)数据竞争（多线程）GCC, Clang~10x 减速
MemorySanitizer (MSan)使用未初始化的内存Clang only~3x 减速

在 CMake 中启用

CMakeLists.txt
# 在 Debug 模式下自动启用 AddressSanitizer
if(CMAKE_BUILD_TYPE STREQUAL "Debug")
    if(CMAKE_CXX_COMPILER_ID MATCHES "Clang|GNU")
        target_compile_options(myapp PRIVATE -fsanitize=address -fno-omit-frame-pointer)
        target_link_options(myapp PRIVATE -fsanitize=address)
    elseif(MSVC)
        # MSVC 从 VS2019 16.9 开始支持 ASan
        target_compile_options(myapp PRIVATE /fsanitize=address)
    endif()
endif()
强烈建议

在 CI/CD 的 Debug 测试中始终启用 ASan 和 UBSan。Google 的工程实践是：所有 C++ 代码在提交前必须通过 ASan 和 TSan 检测。这个习惯能捕获 80% 以上的内存相关 bug。

行业故事：AddressSanitizer 的前世今生

AddressSanitizer（ASan）于 2011 年由 Google 工程师 Konstantin Serebryany 开发。灵感来自一个痛苦的现实：Google 的 C++ 代码库中有数百万行代码，而传统的内存检测工具（如 Valgrind）太慢——会让程序减速 20-50 倍，根本无法在日常测试中使用。

ASan 的核心创新是"shadow memory"——用一块辅助内存来记录每个字节的可访问性。编译器在每次内存访问时插入检查代码，开销只有约 2 倍。这个速度足够快，可以在每次 CI 测试中运行。

ASan 上线后，Google 内部立即发现了数千个潜伏多年的内存 bug——有些已经在生产环境中"安静地"运行了数年，只是碰巧没有触发可见的崩溃。Chrome 团队报告说，ASan 帮助他们修复的 bug 比所有其他方法加起来还多。如今 ASan 已经是 C++ 工具链中不可或缺的一部分。

验证

环境验证脚本

以下是一个完整的环境验证脚本，运行它会检查所有工具是否安装正确：

PowerShell — verify-env.ps1
# ======================================
# C++ 跨平台开发环境验证脚本
# ======================================

Write-Host "`n====== C++ Cross-Platform Dev Environment Check ======" -ForegroundColor Cyan

$errors = @()
$warnings = @()

# 1. 检查 Git
Write-Host "`n[1/8] Git..." -NoNewline
try {
    $gitVer = (git --version 2>&1) -replace 'git version ', ''
    Write-Host " OK ($gitVer)" -ForegroundColor Green
} catch {
    Write-Host " MISSING" -ForegroundColor Red
    $errors += "Git not found"
}

# 2. 检查 CMake
Write-Host "[2/8] CMake..." -NoNewline
try {
    $cmakeVer = (cmake --version 2>&1 | Select-Object -First 1) -replace 'cmake version ', ''
    Write-Host " OK ($cmakeVer)" -ForegroundColor Green
} catch {
    Write-Host " MISSING" -ForegroundColor Red
    $errors += "CMake not found"
}

# 3. 检查 Ninja
Write-Host "[3/8] Ninja..." -NoNewline
try {
    $ninjaVer = ninja --version 2>&1
    Write-Host " OK ($ninjaVer)" -ForegroundColor Green
} catch {
    Write-Host " MISSING" -ForegroundColor Red
    $errors += "Ninja not found"
}

# 4. 检查 MSVC (cl.exe)
Write-Host "[4/8] MSVC..." -NoNewline
$vsWhere = "${env:ProgramFiles(x86)}\Microsoft Visual Studio\Installer\vswhere.exe"
if (Test-Path $vsWhere) {
    $vsPath = & $vsWhere -latest -property installationPath
    Write-Host " OK (VS at $vsPath)" -ForegroundColor Green
} else {
    Write-Host " NOT FOUND" -ForegroundColor Yellow
    $warnings += "Visual Studio not detected"
}

# 5. 检查 GCC (MinGW)
Write-Host "[5/8] GCC (MinGW)..." -NoNewline
try {
    $gccVer = (g++ --version 2>&1 | Select-Object -First 1)
    Write-Host " OK ($gccVer)" -ForegroundColor Green
} catch {
    Write-Host " SKIP (optional)" -ForegroundColor Yellow
    $warnings += "GCC (MinGW) not found - needed for cross-compiler testing"
}

# 6. 检查 Clang
Write-Host "[6/8] Clang..." -NoNewline
try {
    $clangVer = (clang++ --version 2>&1 | Select-Object -First 1)
    Write-Host " OK ($clangVer)" -ForegroundColor Green
} catch {
    Write-Host " SKIP (optional)" -ForegroundColor Yellow
    $warnings += "Clang not found - recommended for clangd and tools"
}

# 7. 检查 vcpkg
Write-Host "[7/8] vcpkg..." -NoNewline
if (Test-Path "C:\vcpkg\vcpkg.exe") {
    $vcpkgVer = (C:\vcpkg\vcpkg.exe version 2>&1 | Select-Object -First 1)
    Write-Host " OK ($vcpkgVer)" -ForegroundColor Green
} elseif (Get-Command vcpkg -ErrorAction SilentlyContinue) {
    Write-Host " OK (found in PATH)" -ForegroundColor Green
} else {
    Write-Host " SKIP (optional)" -ForegroundColor Yellow
    $warnings += "vcpkg not found"
}

# 8. 检查 WSL
Write-Host "[8/8] WSL2..." -NoNewline
try {
    $wslStatus = wsl --status 2>&1
    $wslList = wsl -l -v 2>&1
    Write-Host " OK" -ForegroundColor Green
} catch {
    Write-Host " SKIP (optional)" -ForegroundColor Yellow
    $warnings += "WSL2 not found - needed for Linux target testing"
}

# 汇总
Write-Host "`n====== Summary ======" -ForegroundColor Cyan
if ($errors.Count -eq 0 -and $warnings.Count -eq 0) {
    Write-Host "All checks passed! Environment is ready." -ForegroundColor Green
} elseif ($errors.Count -eq 0) {
    Write-Host "Core tools OK. Warnings:" -ForegroundColor Yellow
    $warnings | ForEach-Object { Write-Host "  - $_" -ForegroundColor Yellow }
} else {
    Write-Host "Errors found:" -ForegroundColor Red
    $errors | ForEach-Object { Write-Host "  - $_" -ForegroundColor Red }
    $warnings | ForEach-Object { Write-Host "  - $_ (warning)" -ForegroundColor Yellow }
}

# 编译测试
Write-Host "`n====== Compile Test ======" -ForegroundColor Cyan
$testDir = "$env:TEMP\cpp-env-test"
New-Item -ItemType Directory -Force -Path $testDir | Out-Null
Set-Content "$testDir\main.cpp" @'
#include <iostream>
#include <vector>
#include <algorithm>
#include <string>

int main() {
    std::vector<std::string> items = {"MSVC", "CMake", "Ninja"};
    std::sort(items.begin(), items.end());
    for (const auto& item : items)
        std::cout << "  OK: " << item << std::endl;
    std::cout << "C++20 structured binding: ";
    auto [a, b] = std::pair{42, 3.14};
    std::cout << a << ", " << b << std::endl;
    return 0;
}
'@

Set-Content "$testDir\CMakeLists.txt" @'
cmake_minimum_required(VERSION 3.20)
project(EnvTest LANGUAGES CXX)
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
add_executable(envtest main.cpp)
'@

Push-Location $testDir
cmake -B build -G Ninja 2>&1 | Out-Null
cmake --build build 2>&1 | Out-Null
if (Test-Path "build\envtest.exe") {
    Write-Host "Compile test:" -NoNewline
    Write-Host " PASSED" -ForegroundColor Green
    & .\build\envtest.exe
} else {
    Write-Host "Compile test:" -NoNewline
    Write-Host " FAILED" -ForegroundColor Red
}
Pop-Location
Remove-Item -Recurse -Force $testDir -ErrorAction SilentlyContinue

将以上内容保存为 verify-env.ps1，然后在 PowerShell 中运行：

常见问题排查

Q1: cl 命令找不到

原因：没有在"开发者命令提示符"中运行。解决：在开始菜单搜索 x64 Native Tools Command Prompt for VS 2022，或者在普通 PowerShell 中运行：

PowerShell
& "C:\Program Files\Microsoft Visual Studio\2022\Community\Common7\Tools\Launch-VsDevShell.ps1" -Arch amd64

Q2: CMake 报 "No CMAKE_CXX_COMPILER could be found"

原因：CMake 找不到编译器。如果使用 MinGW，确保 C:\msys64\ucrt64\bin 在 PATH 中，并且在 CMake 命令中指定生成器：

Shell
cmake -B build -G "MinGW Makefiles"

Q3: vcpkg 安装的库在 CMake 中找不到

确保 CMake 使用了 vcpkg 的 toolchain 文件：

Shell
cmake -B build -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake

Q4: WSL2 中访问 Windows 文件很慢

WSL2 访问 /mnt/c/（Windows 文件系统）的 I/O 性能较差。建议将项目文件放在 WSL2 的 Linux 文件系统中（如 ~/projects/），而不是 /mnt/c/Users/ 下。

Q5: 编译时提示 "C++20 特性不可用"

确保 CMakeLists.txt 中设置了正确的 C++ 标准：

CMakeLists.txt
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)  # 禁用编译器特定扩展

Q6: clangd 不工作或智能补全不准确

clangd 需要 compile_commands.json 文件来理解项目结构。在 CMake 中启用：

Shell
cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

# 或添加到 CMakeLists.txt：
# set(CMAKE_EXPORT_COMPILE_COMMANDS ON)

# 然后在项目根目录创建符号链接：
cp build/compile_commands.json .
