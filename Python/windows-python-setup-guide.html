为什么要用这套环境？

你的核心需求可以归纳为三类：

  • 模型转换：用 PyTorch 训练或加载模型，导出为 ONNX 格式，供其他框架或边缘设备部署。
  • 模型验证：下载社区预训练权重，快速跑通推理，确认模型能力是否符合预期。
  • 原型验证：用 YOLO 系列做目标检测原型，验证数据、调参、评估指标。

这份指南不追求"最全"，而是追求"最顺 + 最对"——每一步都给出 Windows 下最稳的安装路径、被业界认可的工程实践，以及出问题时最常见的排查方法。

文档定位：安装教程 → 工程手册

市面上的 Python 安装教程多如牛毛，但大多停留在"能跑起来"的层面。真正参与过生产级深度学习项目的人都知道，环境管理、代码质量、版本控制、部署优化才是决定项目能否长期维护的关键。本文档从"安装"延伸到"工程化"，覆盖以下维度：

工具链现代化

uv / Poetry 替代裸 pip，ruff / pre-commit 保障代码质量，pyproject.toml 统一配置。

框架选型有据

PyTorch vs TensorFlow vs JAX，不是"谁火用谁"，而是根据场景、团队、部署目标做理性选择。

部署不再踩坑

TensorRT、OpenVINO、量化、服务化，从"模型能跑"到"线上能扛"。

1安装 Python

1.1 下载官方 Python

推荐从 python.org 下载最新稳定版（当前建议 3.11.x）。

为什么不是 3.12？ 部分 PyTorch 的预览功能或第三方库对 3.12 的适配可能滞后。3.11 在性能与兼容性之间是目前的甜点版本。如果你的目标库已明确支持 3.12，也可以上。

安装时务必勾选：

  • Add python.exe to PATH（添加到环境变量）
  • Use admin privileges when installing py.exe
  • Disable path length limit（Windows 长路径支持，避免后续深度学习库解压报错）

安装完成后，打开 PowerShell 或 CMD 验证：

python --version
pip --version

1.2 多版本管理（进阶）：pyenv-win

如果你需要在 3.10 / 3.11 / 3.12 之间切换（比如不同项目依赖不同版本），pyenv-win 是 Windows 下最干净的多版本管理方案。

# 通过 winget 安装（Windows 10 1809+ 自带）
winget install pyenv-win

# 重启终端后
pyenv install 3.11.9
pyenv install 3.12.4
pyenv global 3.11.9

# 在项目目录下锁定版本
pyenv local 3.12.4
pyenv vs 官方安装包？
官方安装包适合"一台机器一个版本"的简单场景；pyenv 适合开发者需要频繁切换版本、或同时维护多个遗留项目的场景。初学者先用官方安装包即可，遇到版本冲突再上 pyenv。

1.3 确认 pip 和基本工具

python -m pip install --upgrade pip setuptools wheel

这三个是基础中的基础：setuptools 负责包的构建，wheel 让安装预编译包时速度翻倍。

2现代 Python 包管理工具

如果你还在用裸 pip install 管理项目依赖，是时候升级了。现代 Python 包管理工具解决了 pip 的核心痛点：没有可靠的依赖锁定、虚拟环境管理繁琐、缺乏可复现的构建声明。

2.1 工具对比：uv vs Poetry vs PDM vs Conda

工具 核心优势 适用场景 学习曲线
uv Rust 编写，速度极快（比 pip 快 10~100 倍）；兼容 pip 命令；内置虚拟环境管理 追求极致速度、CI/CD 场景、替代 pip 的零成本升级 低
Poetry 依赖锁定严格（poetry.lock）；语义版本约束清晰；发布到 PyPI 一站式 中大型项目、团队协作、需要严格可复现构建 中
PDM PEP 582 本地包目录支持；与标准工具链兼容最好；插件生态丰富 喜欢标准规范、需要灵活扩展的开发者 中
Conda 非 Python 依赖管理（CUDA、MKL、OpenCV C++ 等）；科学计算生态成熟 数据科学、需要管理 CUDA/cuDNN 版本、Anaconda 用户 中
推荐选择：
  • 个人快速项目 / CI 场景 → uv（2024~2025 年社区增长最快，Astral 出品，Ruff 同门）
  • 团队协作 / 需要严格锁定 → Poetry（生态最成熟，文档最完善）
  • 数据科学 / CUDA 版本管理 → Conda 或 micromamba（Conda 的 C++ 依赖管理能力目前仍不可替代）

2.2 uv 实战（推荐入门）

# 安装 uv
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# 创建虚拟环境 + 安装包（一步完成，速度极快）
uv venv myenv
uv pip install torch torchvision --index-url https://download.pytorch.org/whl/cu124

# 导出精确锁定文件
uv pip compile pyproject.toml -o requirements.lock
uv pip sync requirements.lock

# uv 的 venv 激活方式与标准 venv 相同
myenv\Scripts\Activate.ps1

2.3 Poetry 实战（团队推荐）

# 安装 Poetry（安装到用户目录，不影响系统 Python）
(Invoke-WebRequest -Uri https://install.python-poetry.org -UseBasicParsing).Content | py -

# 在项目根目录初始化
poetry init

# 添加依赖（自动解析兼容版本并写入 pyproject.toml）
poetry add torch torchvision --source pytorch-cuda
poetry add ultralytics opencv-python

# 安装锁定依赖（所有团队成员拿到同一份 poetry.lock）
poetry install

# 进入 Poetry 管理的虚拟环境
poetry shell

# 运行命令而不进入 shell
poetry run python train.py

2.4 Conda / Miniconda（数据科学场景）

# 推荐安装 Miniconda（精简版，无臃肿预装包）
# 官网：https://docs.conda.io/en/latest/miniconda.html

# 创建隔离环境，同时指定 Python 和 CUDA 工具包版本
conda create -n dl python=3.11 pytorch torchvision cuda-toolkit=12.1 -c pytorch -c nvidia
conda activate dl

# 导出环境（包含非 Python 依赖）
conda env export > environment.yml
conda env create -f environment.yml
conda 与 pip 混用的坑
同一个环境里，优先用 conda 安装底层依赖（numpy、CUDA、MKL），再用 pip 安装上层 Python 包。不要反过来——pip 安装的包可能覆盖 conda 管理的共享库，导致不可预知的冲突。

2.5 pyproject.toml：现代 Python 项目的统一入口

无论用 Poetry、PDM 还是 setuptools，pyproject.toml 都已成为 Python 项目的标准配置文件。它替代了传统的 setup.py、requirements.txt、setup.cfg，把所有配置集中在一处。

[project]
name = "my-dl-project"
version = "0.1.0"
description = "PyTorch model training and deployment pipeline"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.1.0",
    "torchvision>=0.16.0",
    "onnx>=1.15.0",
    "onnxruntime-gpu>=1.16.0",
    "ultralytics>=8.0.0",
    "opencv-python>=4.8.0",
]

[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy", "pre-commit"]

[tool.ruff]
line-length = 100
target-version = "py311"

[tool.mypy]
python_version = "3.11"
warn_return_any = true
warn_unused_configs = true

3项目工程规范

写深度学习代码不等于"在 Jupyter Notebook 里调参"。一旦项目进入团队协作或生产阶段，目录结构、代码风格、版本控制、自动化检查就都变得至关重要。

3.1 推荐目录结构（src layout）

Python 社区有两个主流项目布局：flat layout（源码在根目录）和 src layout（源码在 src/ 下）。深度学习项目强烈推荐 src layout，因为它能避免测试时误导入未安装的开发版本，也更适合 Docker 构建。

my-dl-project/
├── src/
│ └── my_project/
│ ├── __init__.py
│ ├── data/ # 数据加载、预处理、增强
│ │ ├── __init__.py
│ │ └── dataset.py
│ ├── models/ # 模型定义、自定义层
│ │ ├── __init__.py
│ │ └── classifier.py
│ ├── training/ # 训练循环、优化器、学习率调度
│ │ ├── __init__.py
│ │ └── trainer.py
│ ├── export/ # ONNX / TorchScript 导出逻辑
│ │ └── onnx_export.py
│ ├── inference/ # 推理封装、预处理/后处理
│ │ └── predictor.py
│ └── utils/ # 日志、配置加载、通用工具
│ └── logger.py
├── configs/ # Hydra / YAML 配置文件
│ ├── train.yaml
│ └── model/
│ └── resnet50.yaml
├── notebooks/ # 探索性分析（不存大文件）
│ └── eda.ipynb
├── scripts/ # 一次性脚本（训练、导出、评估）
│ ├── train.py
│ ├── export_onnx.py
│ └── evaluate.py
├── tests/ # pytest 测试用例
│ ├── test_dataset.py
│ └── test_model.py
├── docker/ # Dockerfile 及相关配置
│ └── Dockerfile
├── pyproject.toml # 项目元数据 + 工具配置
├── README.md
├── .gitignore
├── .pre-commit-config.yaml
└── data/ # 数据目录（gitignore，不提交）
├── raw/
├── processed/
└── external/
为什么要把源码放在 src/ 下？
在 flat layout 中，import my_project 会直接导入当前目录下的文件夹，即使你还没有 pip install -e . 安装它。这会导致"测试通过了但安装后却报错"的诡异问题。src layout 强迫你必须先安装才能导入，提前暴露安装问题。

4深度学习框架选型指南

选框架不是选信仰，而是根据项目阶段、团队背景、部署目标做理性决策。2024~2025 年的框架格局已相对清晰，但仍各有优劣。

4.1 三大主流框架全景对比

维度 PyTorch 2.x TensorFlow 2.x + Keras JAX + Flax/Haiku
开发范式 动态图（eager），调试直观，类似 Python 动态 + 静态图混合，Keras API 高阶封装 纯函数式，向量化映射（vmap/pmap），函数变换
学习曲线 平缓，文档友好 中等，Keras 简单但底层复杂 陡峭，函数式思维需适应
调试体验 极佳，pdb 原生支持 可接受，tf.function 调试较痛苦 一般，JIT 编译错误信息晦涩
研究生态 最强，论文复现首选 较强，工业界模型多 增长快，Google DeepMind 主力
生产部署 TorchScript / torch.export / ONNX TF SavedModel / TF Lite / TF Serving 成熟 较弱，主要面向研究
移动端/边缘 PyTorch Mobile（一般），需转 ONNX/TF Lite TF Lite 业界标准 基本不支持
性能优化 torch.compile（2.0+）极大提升训练推理速度 XLA / Grappler 优化成熟 XLA 编译优化极强
Windows 支持 一等公民，官方 wheel 完善 官方支持 Linux 为主，Windows 支持弱

4.2 选型决策树

  • 你是深度学习初学者 → PyTorch（教程最多、社区最大、调试最友好）
  • 你要复现最新论文 → PyTorch（95%+ 的论文开源代码是 PyTorch）
  • 你要部署到 Android/iOS 嵌入式设备 → TensorFlow + TF Lite（移动端生态最强）
  • 你在 Google Cloud / TPU 上训练 → JAX 或 TensorFlow（TPU 对 PyTorch 支持仍不如 TF/JAX）
  • 你需要大规模分布式训练（千张 GPU） → PyTorch + FSDP，或 JAX + pmap（两者都很强）
  • 你在 Windows 上做开发 → PyTorch（JAX 在 Windows 上支持极差，TensorFlow 安装较臃肿）

4.3 PyTorch 2.x 核心新特性

PyTorch 2.0 不是小版本更新，它引入了 torch.compile，通过图编译技术在不改动模型代码的情况下大幅提升性能。

import torch

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters())

# 用 torch.compile 包裹模型，自动图优化
model = torch.compile(model, mode="default")
# mode 可选："default"（推荐）、"reduce-overhead"（推理）、"max-autotune"（极致性能）

# 训练代码完全不变
for data, target in dataloader:
    optimizer.zero_grad()
    output = model(data)
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()
torch.compile 实际收益：在 A100 / RTX 4090 上，典型 CV 模型训练速度提升 30%~70%，且无需改模型代码。Windows 下同样可用，但 Linux 上的 CUDA graph 支持更完整。

5安装 PyTorch

PyTorch 是整个工具链的核心。安装方式取决于你是否有 NVIDIA 独立显卡。

5.1 检查你的硬件

打开任务管理器 → 性能 → GPU，如果有 NVIDIA GPU，记住型号，然后去 CUDA GPU 列表 确认是否支持 CUDA。

5.2 有 NVIDIA 显卡（推荐 GPU 版）

先安装 NVIDIA 显卡驱动（GeForce Experience 或官网驱动即可），确保驱动较新。PyTorch 2.x 通常需要 CUDA 11.8 或 12.x。

去 pytorch.org 选择对应配置，生成命令。以下是目前最常用的两条：

# CUDA 12.4（较新的 40/50 系显卡推荐）
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu124

# CUDA 11.8（兼容性更广，30/20 系也适用）
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu118

5.3 只有核显 / 纯 CPU

pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cpu

5.4 验证安装

python -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'CPU only')"

如果输出 True 和显卡名称，恭喜，GPU 加速就绪。

torchvision / torchaudio 是干什么的？
torchvision 提供图像预处理、常见模型（ResNet、MobileNet 等）和数据集工具；torchaudio 处理音频。如果你只做图像/模型转换，torchaudio 可以省略，但一起装也没坏处。

5.5 推荐配套安装：timm 和 Hugging Face Hub

# timm：PyTorch 图像模型库，提供 900+ 预训练模型
pip install timm

# Hugging Face Hub：下载社区预训练模型
pip install huggingface_hub

6安装 ONNX 工具链

ONNX（Open Neural Network Exchange）是模型跨框架迁移的桥梁。PyTorch 原生支持导出 ONNX，但你需要 surrounding tools 来验证、简化、优化导出的模型。

6.1 核心包

pip install onnx onnxruntime onnxscript
包名作用
onnxONNX 格式定义和基础工具，读写 .onnx 文件
onnxruntime微软开发的高性能推理引擎，支持 CPU/GPU，验证模型是否可正常推理
onnxscript用 Python 直接编写/转换 ONNX 图，调试时很有用

6.2 GPU 推理加速（可选）

如果你要在本机用 GPU 跑 ONNX 推理，安装 GPU 版 Runtime：

pip install onnxruntime-gpu
注意：onnxruntime 和 onnxruntime-gpu 不要同时安装，会冲突。如果已经装了 CPU 版，先卸载再装 GPU 版。

6.3 快速验证 ONNX 导出

import torch
import onnx

# 创建一个 dummy 模型
class DummyModel(torch.nn.Module):
    def forward(self, x):
        return x * 2

model = DummyModel()
dummy_input = torch.randn(1, 3, 224, 224)

# 导出
torch.onnx.export(
    model, dummy_input, "dummy.onnx",
    input_names=["input"], output_names=["output"],
    opset_version=17
)

# 验证
onnx_model = onnx.load("dummy.onnx")
onnx.checker.check_model(onnx_model)
print("ONNX 模型验证通过！")

# 用 onnxruntime 跑推理
import onnxruntime as ort
session = ort.InferenceSession("dummy.onnx")
outputs = session.run(None, {"input": dummy_input.numpy()})
print(outputs[0].shape)

# 模型简化（去除冗余节点，减小体积）
pip install onnxsim

6.5 ONNX Runtime 执行提供者（Execution Providers）

ONNX Runtime 的强大之处在于支持多种后端（Execution Provider），同一模型可以在不同硬件上加速运行：

执行提供者适用硬件说明
CUDAExecutionProviderNVIDIA GPUNVIDIA GPU 推理，需 onnxruntime-gpu
TensorrtExecutionProviderNVIDIA GPUNVIDIA TensorRT 加速，性能最佳
DmlExecutionProviderWindows DirectML支持 NVIDIA/AMD/Intel GPU，Windows 友好
OpenVINOExecutionProviderIntel CPU/GPU/NPUIntel 硬件最优，需 openvino
CPUExecutionProvider通用 CPU默认回退，无额外依赖
import onnxruntime as ort

# 显式指定 CUDA 后端

session = ort.InferenceSession("model.onnx", providers=providers)

# 查看当前可用的 providers
print(ort.get_available_providers())

9生产级部署与优化

模型训练完成只是 halfway。要把模型搬到生产环境，你需要面对一系列实际问题：推理延迟、吞吐量、内存占用、硬件兼容性、服务化接口。本节覆盖业界主流的部署优化工具链。

9.1 部署路径总览

通用决策流程：

  1. 目标平台是 云端服务器 / 工作站（NVIDIA GPU）→ TensorRT 或 ONNX Runtime + CUDA
  2. 目标平台是 Intel CPU / 集成显卡 / NPU → OpenVINO
  3. 目标平台是 手机 / 嵌入式 / ARM → TensorFlow Lite 或 ONNX Runtime Mobile
  4. 目标平台是 浏览器 / Web → ONNX Runtime Web 或 TensorFlow.js
  5. 需要 跨平台统一方案 → ONNX 作为中间格式，再转各平台运行时

9.2 TensorRT：NVIDIA GPU 推理的终极加速

TensorRT 是 NVIDIA 的高性能深度学习推理 SDK，通过层融合、精度校准、内核自动调优，通常能将 PyTorch/ONNX 模型的推理速度提升 2~10 倍。

安装

# 方式一：通过 pip（Python API，推荐）
pip install tensorrt

# 方式二：通过 NVIDIA 官方安装包（包含 trtexec 等命令行工具）
# 下载地址：https://developer.nvidia.com/tensorrt

# 验证安装
python -c "import tensorrt as trt; print(trt.__version__)"

从 ONNX 构建 TensorRT 引擎

import tensorrt as trt

logger = trt.Logger(trt.Logger.WARNING)
builder = trt.Builder(logger)
network = builder.create_network(1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH))
parser = trt.OnnxParser(network, logger)

# 解析 ONNX
with open("model.onnx", "rb") as f:
    parser.parse(f.read())

# 配置 builder
config = builder.create_builder_config()
config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 << 30)  # 1GB

# FP16 精度（速度和内存减半，精度损失通常极小）
config.set_flag(trt.BuilderFlag.FP16)

# 构建引擎
engine = builder.build_engine(network, config)

# 序列化保存（下次直接加载，无需重新构建）
with open("model.trt", "wb") as f:
    f.write(engine.serialize())
TensorRT 的局限性：
  • 仅支持 NVIDIA GPU，无法用于 AMD/Intel 显卡或 CPU
  • 引擎文件与 GPU 架构绑定（在 RTX 4090 上构建的引擎不能在 A100 上直接用）
  • 部分 ONNX 算子可能不支持，需要用 polygraphy 或 onnx-graphsurgeon 修改图

9.3 OpenVINO：Intel 硬件的最优解

如果你的部署目标是 Intel CPU、集成显卡、NPU（如 Intel Core Ultra 的 AI Boost），OpenVINO 是最佳选择。它也支持部分 ARM 和 NVIDIA GPU。

# 安装 OpenVINO
pip install openvino

# 将 ONNX 模型转为 OpenVINO IR 格式
import openvino as ov

core = ov.Core()
model = core.read_model("model.onnx")
compiled_model = core.compile_model(model, "AUTO")  # AUTO 自动选择最佳设备

# 推理
input_tensor = compiled_model.input(0)
output_tensor = compiled_model.output(0)
infer_request = compiled_model.create_infer_request()
infer_request.infer({input_tensor: your_numpy_array})
result = infer_request.get_output_tensor(output_tensor).data

9.4 模型量化：用精度换速度

量化是将模型权重和激活值从高精度（FP32）降到低精度（FP16、INT8、甚至 INT4）的技术，能显著减少模型体积和推理延迟。

量化类型精度体积减少速度提升精度损失工具
FP1616-bit 浮点~50%1.5~2x极小TensorRT、ONNX Runtime
动态 INT88-bit 整数~75%2~4x小ONNX Runtime、PyTorch Quantization
静态 INT8（PTQ）8-bit 整数~75%3~8x中TensorRT、OpenVINO
QAT（量化感知训练）8-bit 整数~75%3~8x极小PyTorch FX、TensorFlow

PyTorch 动态量化示例（最简单，适合 CPU 推理）

import torch.quantization

model = MyModel()
model.load_state_dict(torch.load("model.pth"))
model.eval()

# 动态量化（只对 Linear/LSTM 层生效，适合 CPU）
quantized_model = torch.quantization.quantize_dynamic(
    model, {torch.nn.Linear}, dtype=torch.qint8
)

# 保存量化模型
torch.save(quantized_model.state_dict(), "model_quantized.pth")

# 量化后模型在 CPU 上推理速度通常提升 2~4 倍

9.6 服务化部署：FastAPI vs Triton Inference Server

方案适用场景复杂度性能
FastAPI + ONNX Runtime中小规模、快速上线、REST API低中（单进程，可用 Gunicorn 多 worker）
Triton Inference Server大规模生产、高并发、模型版本管理、多框架支持高高（动态批处理、并发执行、GPU 共享）
BentoML / MLflowMLOps 全链路、模型注册、A/B 测试中中

FastAPI 最小服务化示例

# 安装：pip install fastapi uvicorn python-multipart
from fastapi import FastAPI, File, UploadFile
import numpy as np
import onnxruntime as ort
from PIL import Image
import io

app = FastAPI(title="Image Classification API")
session = ort.InferenceSession("model.onnx")
input_name = session.get_inputs()[0].name

@app.post("/predict")
async def predict(file: UploadFile = File(...)):
    image = Image.open(io.BytesIO(await file.read())).convert("RGB")
    image = image.resize((224, 224))
    arr = np.array(image).astype(np.float32) / 255.0
    arr = np.transpose(arr, (2, 0, 1))  # HWC -> CHW
    arr = np.expand_dims(arr, 0)        # add batch dim

    outputs = session.run(None, {input_name: arr})
    return {"prediction": outputs[0].tolist()}

# 启动：uvicorn server:app --host 0.0.0.0 --port 8000

src/cifar10_project/model.py

import torch
import torch.nn as nn
import torchvision.models as models


def create_model(num_classes: int = 10, pretrained: bool = False) -> nn.Module:
    """创建适配 CIFAR-10 的 ResNet-18。

    CIFAR-10 图像是 32x32，原版 ResNet 针对 224x224 设计，
    因此我们将首层 conv 的 kernel_size 从 7 改为 3，stride 从 2 改为 1。
    """
    model = models.resnet18(weights="IMAGENET1K_V1" if pretrained else None)

    # 修改首层 conv 适配 32x32 输入
    model.conv1 = nn.Conv2d(3, 64, kernel_size=3, stride=1, padding=1, bias=False)
    model.maxpool = nn.Identity()  # CIFAR-10 不需要大的下采样

    # 修改最后一层全连接
    model.fc = nn.Linear(model.fc.in_features, num_classes)
    return model

scripts/train.py

import torch
import torch.nn as nn
from torch.utils.data import DataLoader
from torchvision import datasets, transforms
from tqdm import tqdm

from cifar10_project.model import create_model


def main() -> None:
    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

    # 数据增强：训练时做随机裁剪和翻转，测试时只归一化
    train_transform = transforms.Compose([
        transforms.RandomCrop(32, padding=4),
        transforms.RandomHorizontalFlip(),
        transforms.ToTensor(),
        transforms.Normalize((0.4914, 0.4822, 0.4465), (0.2470, 0.2435, 0.2616)),
    ])
    test_transform = transforms.Compose([
        transforms.ToTensor(),
        transforms.Normalize((0.4914, 0.4822, 0.4465), (0.2470, 0.2435, 0.2616)),
    ])

    train_ds = datasets.CIFAR10(root="./data", train=True, download=True, transform=train_transform)
    test_ds = datasets.CIFAR10(root="./data", train=False, download=True, transform=test_transform)

    train_loader = DataLoader(train_ds, batch_size=128, shuffle=True, num_workers=4, pin_memory=True)
    test_loader = DataLoader(test_ds, batch_size=256, shuffle=False, num_workers=4, pin_memory=True)

    model = create_model(num_classes=10, pretrained=False).to(device)
    # PyTorch 2.0+ 编译加速
    if hasattr(torch, "compile"):
        model = torch.compile(model)

    criterion = nn.CrossEntropyLoss()
    optimizer = torch.optim.SGD(model.parameters(), lr=0.1, momentum=0.9, weight_decay=5e-4)
    scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=200)

    best_acc = 0.0
    for epoch in range(200):
        model.train()
        for images, labels in tqdm(train_loader, desc=f"Epoch {epoch+1}"):
            images, labels = images.to(device), labels.to(device)
            optimizer.zero_grad()
            outputs = model(images)
            loss = criterion(outputs, labels)
            loss.backward()
            optimizer.step()
        scheduler.step()

        # 验证
        model.eval()
        correct = total = 0
        with torch.no_grad():
            for images, labels in test_loader:
                images, labels = images.to(device), labels.to(device)
                outputs = model(images)
                _, predicted = outputs.max(1)
                total += labels.size(0)
                correct += predicted.eq(labels).sum().item()

        acc = 100.0 * correct / total
        print(f"Epoch {epoch+1}: Accuracy = {acc:.2f}%")
        if acc > best_acc:
            best_acc = acc
            torch.save(model.state_dict(), "best_model.pth")

    print(f"Best accuracy: {best_acc:.2f}%")


if __name__ == "__main__":
    main()

server.py（FastAPI 服务）

from fastapi import FastAPI, File, UploadFile
import numpy as np
import onnxruntime as ort
from PIL import Image
import io

app = FastAPI(title="CIFAR-10 Classification")

# 使用 CUDA 加速（如果可用）
providers = ["CUDAExecutionProvider", "CPUExecutionProvider"]
session = ort.InferenceSession("cifar10_resnet18_sim.onnx", providers=providers)
input_name = session.get_inputs()[0].name

# CIFAR-10 归一化参数
MEAN = np.array([0.4914, 0.4822, 0.4465])
STD = np.array([0.2470, 0.2435, 0.2616])
CLASSES = ["airplane", "automobile", "bird", "cat", "deer",
           "dog", "frog", "horse", "ship", "truck"]


@app.post("/predict")
async def predict(file: UploadFile = File(...)):
    image = Image.open(io.BytesIO(await file.read())).convert("RGB")
    image = image.resize((32, 32))
    arr = np.array(image).astype(np.float32) / 255.0
    arr = (arr - MEAN) / STD
    arr = np.transpose(arr, (2, 0, 1))
    arr = np.expand_dims(arr, 0)

    outputs = session.run(None, {input_name: arr})[0]
    pred_idx = int(np.argmax(outputs))
    confidence = float(np.max(outputs))

    return {
        "class": CLASSES[pred_idx],
        "confidence": round(confidence, 4),
        "all_scores": {cls: round(float(score), 4) for cls, score in zip(CLASSES, outputs[0])}
    }


@app.get("/health")
async def health():
    return {"status": "ok", "providers": ort.get_available_providers()}

    logger = logging.getLogger(name)
    logger.setLevel(level)

    if not logger.handlers:
        handler = logging.StreamHandler(sys.stdout)
        formatter = logging.Formatter(
            "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s",
            datefmt="%Y-%m-%d %H:%M:%S",
        )
        handler.setFormatter(formatter)
        logger.addHandler(handler)

    return logger


# 使用

logger.info("Training started")
logger.warning("Learning rate decay triggered")
logger.error("Failed to load checkpoint")

11.6 测试策略

深度学习代码也需要测试，至少覆盖以下方面：

  • 模型前向传播：给定固定输入，输出 shape 是否正确
  • 数据加载：Dataset 返回的数据类型、shape、范围是否符合预期
  • ONNX 导出一致性：PyTorch 和 ONNX Runtime 的输出差异是否在容差内
  • 配置文件校验：缺失必填字段时是否抛异常
# tests/test_model.py
import torch
from cifar10_project.model import create_model


def test_model_output_shape() -> None:
    model = create_model(num_classes=10)
    x = torch.randn(4, 3, 32, 32)
    out = model(x)
    assert out.shape == (4, 10), f"Expected (4, 10), got {out.shape}"


def test_model_onnx_consistency() -> None:
    model = create_model(num_classes=10)
    model.eval()
    x = torch.randn(1, 3, 32, 32)

    # PyTorch 输出
    with torch.no_grad():
        pt_out = model(x).numpy()

    # ONNX 输出
    import onnxruntime as ort
    session = ort.InferenceSession("cifar10_resnet18.onnx")
    onnx_out = session.run(None, {"input": x.numpy()})[0]

    assert pt_out.shape == onnx_out.shape
    assert torch.allclose(torch.from_numpy(pt_out), torch.from_numpy(onnx_out), atol=1e-5)

依赖组件的背景故事

知其然，也知其所以然。了解这些工具的来龙去脉，有助于你在遇到问题时判断该往哪个方向排查。

1991 — Python
Guido van Rossum 的"圣诞礼物"
Python 诞生于荷兰的圣诞节假期。Guido 希望创造一种简洁、可读性强的语言，让程序员把精力花在解决问题上，而非语法细节上。30 多年后，Python 凭借 NumPy、SciPy 等科学计算生态，意外成为 AI 时代的第一语言。
2006 — CUDA
NVIDIA 的"通用计算"豪赌
CUDA 最初被游戏玩家和图形工作者忽视，但 NVIDIA 坚持让 GPU 能跑通用程序。当 2012 年 AlexNet 用两块 GTX 580 赢得 ImageNet 后，CUDA 成为深度学习的事实标准。没有 CUDA，现代 AI 的训练周期可能要延长数十倍。
2012 — AlexNet
深度学习的一声惊雷
虽然 AlexNet 本身不是工具，但它证明了 GPU + 深度卷积网络的可行性。从此，模型参数从百万级跃升至亿级，框架之争也拉开序幕：Caffe、Torch、Theano 相继登场。
2016 — PyTorch
Meta（Facebook）的"动态图"逆袭
基于古老的 Torch 框架，Meta 的 AI Research 团队推出了 PyTorch。它以动态计算图（define-by-run）为核心，调试时像写普通 Python 一样直观，迅速在学术界和工业界击败 TensorFlow 成为主流。PyTorch 2.0 引入的 torch.compile 更让它在性能上补足短板。
2017 — ONNX
微软与 Meta 的"停战协议"
深度学习框架割据（TensorFlow、PyTorch、Caffe2、MXNet...），模型互迁成本极高。微软与 Meta 联合发起 ONNX，试图定义一种开放的中间表示格式。如今 ONNX 已成为部署到移动端、边缘设备、甚至 Web 的标配中转格式。
2018 — ONNX Runtime
微软的推理引擎
光有格式不够，还需要一个高性能的"播放器"。ONNX Runtime 由微软开源，支持 CPU、CUDA、DirectML、OpenVINO 等多种后端。它不仅是验证工具，更是许多生产环境（包括 Azure、Office）的实际推理引擎。
2016 / 2023 — YOLO / Ultralytics
从学术论文到工程利器
Joseph Redmon 提出 YOLO 时，目标检测还是以两阶段方法（R-CNN 系列）为主。YOLO 的"单阶段、端到端"思路把速度提升了几个数量级。后来 Glenn Jocher 创办的 Ultralytics 公司推出 YOLOv5/v8/v11，用极其简单的 Python API 把目标检测变成几行代码的事，彻底降低了使用门槛。
2020 — Hugging Face
AI 界的"GitHub"
这个从聊天机器人创业失败转型而来的公司，意外建成了全球最大的预训练模型托管平台。Transformers 库几乎成了 NLP 的标准入口，而 Hugging Face Hub 也汇聚了大量视觉、语音、多模态模型，下载模型再也不是靠百度云盘了。
2023 — uv / Ruff（Astral）
Python 工具链的"Rust 革命"
Astral 公司用 Rust 重写了 Python 工具链：Ruff 替代了 flake8+black+isort，uv 替代了 pip+venv。速度提升 10~100 倍的同时保持了兼容性。这是 Python 生态近年来最重要的基础设施升级之一。

常见问题排查

pip 安装极慢或失败

换国内镜像源：

pip install xxx -i https://pypi.tuna.tsinghua.edu.cn/simple

或配置永久镜像：

pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple

Poetry 用户：

poetry source add --priority=default tsinghua https://pypi.tuna.tsinghua.edu.cn/simple
CUDA out of memory

减小 batch size，或改用 torch.no_grad() 上下文做推理。如果只是想验证模型，CPU 模式完全够用。训练时可用 torch.cuda.empty_cache() 释放碎片。

torch.cuda.is_available() 返回 False
  1. 检查 NVIDIA 驱动是否安装
  2. 检查 PyTorch 的 CUDA 版本是否与系统 CUDA 匹配
  3. 笔记本需确认没开在核显模式
  4. conda 用户确认安装了 cudatoolkit 或 cuda-nvcc
Ultralytics 报错找不到模型

检查当前目录是否有 .pt 文件，或让 YOLO 自动下载：YOLO("yolov8n.pt") 第一次会自动拉取。国内用户可能因网络问题下载失败，可手动从 GitHub Releases 下载。

ONNX 导出后推理结果不对

重点检查 preprocessing（归一化、resize、通道顺序 BGR/RGB）和 postprocessing（NMS、置信度阈值）是否与训练时一致。ONNX 只负责"算"，不负责"预处理"。建议使用同一批数据对比 PyTorch 和 ONNX 的输出差异。

PowerShell 激活环境报错

执行 Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser，或改用 CMD 的 activate.bat。

TensorRT 构建引擎失败

常见原因：ONNX 算子不支持、CUDA 版本不匹配、显存不足。用 polygraphy debug 定位不支持的算子，或用 onnx-graphsurgeon 修改图结构。

uv / Poetry 创建的虚拟环境找不到包

确认你激活了正确的虚拟环境。uv 和 Poetry 的虚拟环境路径与标准 venv 不同（通常在项目根目录的 .venv 下）。用 poetry env info 或 uv pip list 确认当前环境。
