1.1 PyTorch 生态格式详解

PyTorch 原生使用 Python pickle 协议序列化模型，这是最灵活但也最危险的格式：

torch.save 的底层结构
# PyTorch .pth 文件本质是一个 pickle 序列化的 dict
{
    'model_state_dict': OrderedDict([           # 模型参数
        ('conv1.weight', tensor([...])),      # 权重张量
        ('conv1.bias', tensor([...])),        # 偏置张量
        # ...
    ]),
    'optimizer_state_dict': {...},             # 优化器状态（训练用）
    'epoch': 100,                           # 训练轮次
    'loss': 0.0234                          # 损失值
}
安全风险：.pth 基于 pickle，加载时会执行任意 Python 代码。切勿加载不可信来源的 .pth 文件。生产环境应使用 .safetensors 替代。

1.2 TensorFlow 生态格式

SavedModel

目录格式，包含 saved_model.pb（计算图）和 variables/（权重）。TensorFlow 2.x 的标准保存格式，支持签名定义便于 Serving。

Keras H5

单文件 HDF5 格式，存储结构和权重。仅适用于 Keras 模型，不推荐使用于生产部署，建议迁移到 SavedModel。

1.3 通用交换格式：ONNX

ONNX（Open Neural Network Exchange）是跨框架部署的事实标准中间格式。它将计算图序列化为 protobuf，定义了 100+ 标准算子。

ONNX 核心优势：一次转换，多处运行。PyTorch → ONNX → TensorRT / ONNX Runtime / OpenVINO / Core ML / TFLite，实现部署生态解耦。

📱3. 端侧部署实战

端侧部署的核心约束：算力有限、内存紧张、功耗敏感、延迟要求高。必须在精度损失可接受的前提下，最大化压缩模型。

3.1 移动端部署：Android 与 iOS

Android：TensorFlow Lite 方案

TensorFlow Lite 是 Android 端侧推理的首选。Google 提供了完整的工具链：模型转换、量化、基准测试、委托（Delegate）加速。

Android Kotlin — TFLite 推理示例
import org.tensorflow.lite.Interpreter
import org.tensorflow.lite.gpu.GpuDelegate

// 1. 加载模型（assets 或本地文件）
val model = File(context.filesDir, "model.tflite")

// 2. 配置 Interpreter（可选 GPU 加速）
val options = Interpreter.Options().apply {
    // 使用 GPU 委托（Adreno / Mali）
    addDelegate(GpuDelegate())
    // 或使用 NNAPI（Android 8.1+，NPU 加速）
    // setUseNNAPI(true)
    setNumThreads(4)
}

val interpreter = Interpreter(model, options)

// 3. 准备输入（ByteBuffer 或 Multi-dimensional Array）
val inputBuffer = ByteBuffer.allocateDirect(1 * 224 * 224 * 3 * 4)
inputBuffer.order(ByteOrder.nativeOrder())
// ... 填充归一化后的图像数据

// 4. 执行推理
val output = Array(1) { FloatArray(1000) }
interpreter.run(inputBuffer, output)

// 5. 释放资源（重要！）
interpreter.close()

iOS：Core ML 与 TFLite 双轨

Apple 生态优先使用 Core ML，可充分利用 Neural Engine（A11 及以后芯片）。跨平台需求可使用 TFLite iOS。

NNAPI vs GPU Delegate：NNAPI 在搭载 NPU 的新设备上（如 Pixel、Samsung Galaxy）性能优于 GPU Delegate；但在旧设备或 GPU 更强劲的设备上，GPU Delegate 可能反超。建议运行时基准测试决定。

3.2 嵌入式与 IoT 部署

MCU 级设备（STM32、ESP32、Raspberry Pi Pico）资源极度受限，通常使用 TensorFlow Lite for Microcontrollers（TFLM）。

🔄4. 模型格式转换全流程

从训练环境到生产环境的典型路径是：PyTorch (.pth) → ONNX → TFLite / TensorRT / Core ML。本节给出每一步的精确命令和代码。

4.1 PTH → ONNX 转换

这是整个部署链路中最关键的一步。ONNX 作为中间表示，决定了后续所有部署路径的可行性。

export_to_onnx.py — PyTorch 导出 ONNX
import torch
import torch.onnx

# 1. 加载 PyTorch 模型
model = MyModel()
model.load_state_dict(torch.load("model.pth", map_location="cpu"))
model.eval()

# 2. 定义 dummy input（形状必须与真实输入一致）
dummy_input = torch.randn(1, 3, 224, 224)

# 3. 导出 ONNX
torch.onnx.export(
    model,                              # 要导出的模型
    dummy_input,                        # 示例输入
    "model.onnx",                      # 输出路径
    export_params=True,               # 存储权重
    opset_version=17,                 # ONNX 算子集版本（建议 14+）
    do_constant_folding=True,         # 常量折叠优化
    input_names=["input"],             # 输入节点名
    output_names=["output"],           # 输出节点名
    dynamic_axes={                      # 动态维度（批大小可变）
        "input": {0: "batch_size"},
        "output": {0: "batch_size"}
    }
)
常见失败原因：控制流（if/for 依赖张量值）、列表/字典输出、自定义算子、原地操作（in-place ops，如 x += 1）。这些问题在 torch.onnx.export 时会产生警告或错误。

ONNX 模型验证与简化

validate_onnx.py — 验证与简化
import onnx
from onnxsim import simplify

# 1. 加载并检查模型
model = onnx.load("model.onnx")
onnx.checker.check_model(model)          # 抛出异常表示结构有问题

# 2. 使用 onnxsim 简化（消除冗余节点）
model_simp, check = simplify(model)
assert check, "简化后的模型验证失败"
onnx.save(model_simp, "model_simplified.onnx")

# 3. 使用 ONNX Runtime 验证数值一致性
import onnxruntime as ort
import numpy as np

sess = ort.InferenceSession("model_simplified.onnx")
input_name = sess.get_inputs()[0].name
np_input = np.random.randn(1, 3, 224, 224).astype(np.float32)

# PyTorch 结果
with torch.no_grad():
    pt_out = model(torch.from_numpy(np_input)).numpy()

# ONNX 结果
onnx_out = sess.run(None, {input_name: np_input})[0]

# 数值差异检查
np.testing.assert_allclose(pt_out, onnx_out, rtol=1e-5, atol=1e-6)
print("数值一致性验证通过")

4.2 ONNX → TFLite 转换

TFLite 转换通常有两条路径：ONNX → TFLite（直接）或 ONNX → TensorFlow SavedModel → TFLite（间接）。推荐使用 onnx2tf 工具，转换成功率最高。

CLI — 使用 onnx2tf 转换
# 安装转换工具
pip install onnx2tf onnx_graphsurgeon sng4onnx

# 基础转换（含优化）
onnx2tf -i model.onnx -o model_tflite

# 带 INT8 量化（需要代表性数据集）
onnx2tf \
    -i model.onnx \
    -o model_int8.tflite \
    -oiqt \
    -qt per-tensor
Python — 完整量化校准流程
import tensorflow as tf

# 1. 加载 SavedModel（由 onnx2tf 生成）
converter = tf.lite.TFLiteConverter.from_saved_model("model_tflite")

# 2. 配置量化
converter.optimizations = [tf.lite.Optimize.DEFAULT]

# 3. 代表性数据集（用于确定量化范围）
def representative_dataset():
    for image in calibration_dataset:    # 约 100-500 张样本
        image = np.expand_dims(image, axis=0).astype(np.float32)
        yield [image]

converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [
    tf.lite.OpsSet.TFLITE_BUILTINS_INT8    # 强制 INT8
]
converter.inference_input_type = tf.uint8    # 输入 uint8（加速推理）
converter.inference_output_type = tf.float32 # 输出保持 float

# 4. 转换
tflite_model = converter.convert()
with open("model_quantized.tflite", "wb") as f:
    f.write(tflite_model)

4.3 ONNX → TensorRT 转换（GPU 极致优化）

convert_to_trt.py — TensorRT Engine 构建
import tensorrt as trt

TRT_LOGGER = trt.Logger(trt.Logger.WARNING)

def build_engine(onnx_path, engine_path, fp16=True, int8=False):
    builder = trt.Builder(TRT_LOGGER)
    network = builder.create_network(
        1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH)
    )
    parser = trt.OnnxParser(network, TRT_LOGGER)

    with open(onnx_path, "rb") as f:
        if not parser.parse(f.read()):
            for i in range(parser.num_errors):
                print(parser.get_error(i))
            raise RuntimeError("ONNX 解析失败")

    config = builder.create_builder_config()
    config.max_workspace_size = 4 * 1024 * 1024 * 1024  # 4GB

    if fp16:
        config.set_flag(trt.BuilderFlag.FP16)
    if int8:
        config.set_flag(trt.BuilderFlag.INT8)
        # 需配置 INT8 校准器...

    # 构建 serialized engine（耗时操作）
    engine = builder.build_engine(network, config)

    with open(engine_path, "wb") as f:
        f.write(engine.serialize())

build_engine("model.onnx", "model.trt", fp16=True)

⚠5. 模型转换关键注意事项

转换失败是部署工作的最大时间杀手。以下经验来自数十次实际项目的踩坑总结。

5.1 PyTorch → ONNX 阶段

  • opset_version：使用尽可能高的 opset（推荐 14+），低版本可能不支持新算子。
  • 控制流：torch.jit.script 中的控制流可以导出，但 trace 模式下不行。建议先用 torch.jit.script 包装模型。
  • 动态轴：如果批大小固定为 1，不要声明 dynamic_axes，有助于后续 TensorRT 优化。
  • 原地操作：x += 1、x[0] = 1 等原地操作在 ONNX 中可能导致 graph 错误，应替换为 x = x + 1。
  • 列表输出：ONNX 原生不支持 Python list/dict 输出。需改为 tuple 或拼接为单个张量。

5.2 ONNX → TFLite 阶段

  • 算子兼容性：TFLite 的算子集是 ONNX 的子集。TFLite 不支持 NMS（非极大值抑制）、部分激活函数、某些 Padding 模式。
  • 维度对齐：TFLite 偏好 NHWC（通道在后），而 PyTorch/ONNX 默认 NCHW（通道在前）。转换工具会自动插入 Transpose，但可能引入额外开销。
  • 归一化层：BatchNorm 与 Conv 融合（fuse）是常见的图优化，但在量化模型中融合顺序错误会导致精度崩溃。
  • 动态形状：TFLite 对动态形状支持有限。如果模型有变长输入（如文本），需固定为最大长度或按批次拆分。
精度回退陷阱：某些算子在 INT8 量化下精度损失极大（如 Softmax、Sigmoid）。TFLite 的混合量化（部分算子 FP32、部分 INT8）是默认行为，不要强行将所有算子压到 INT8。

5.3 量化特别注意事项

量化类型 压缩比 精度损失 适用场景
FP32 → FP16 2x 几乎无 GPU 推理（TensorRT、Core ML）
FP32 → INT8（PTQ） 4x 低-中（1-3% mAP 下降） 视觉模型、ARM CPU
FP32 → INT8（QAT） 4x 极低 对精度要求高的场景
LLM → INT4/INT5（GPTQ/AWQ） 8x 中（可感知） 大语言模型本地运行
量化经验法则：PTQ（训练后量化）对 CNN 友好，对 Transformer 一般，对 LLM 很差。LLM 量化务必使用 GPTQ、AWQ 或 GGUF 的专用量化方案，而非通用 TFLite 量化。

📋6. Python 代码审查检查清单

在阅读和审查模型相关 Python 代码时，以下检查点可帮助快速定位部署风险。

6.1 模型定义文件（model.py）

  • 输入形状硬编码：检查 forward 中是否有 x.size(0) 决定控制流，这会导致 ONNX 导出失败。
  • device 硬编码：搜索 .cuda()、.to('cuda')，确保导出代码使用 map_location='cpu' 加载。
  • 自定义算子：检查是否有 torch.autograd.Function 或 C++ 扩展，这些通常无法导出。
  • 条件判断：检查 if self.training: 分支，导出时应调用 model.eval() 锁定到推理分支。
  • 归一化位置：确认预处理中的 mean/std 归一化是在 Python 端还是已内嵌到模型中。TFLite 通常希望归一化内嵌。

6.2 导出脚本（export.py）

  • dummy_input 代表性：dummy input 的值范围应覆盖真实输入（如 [0,1] 或 [-1,1]），否则量化校准会出错。
  • torch.no_grad()：导出过程是否包裹在 with torch.no_grad(): 中。
  • 状态一致性：确认导出前调用了 model.eval()，BatchNorm 和 Dropout 行为与推理一致。
  • ONNX 验证：导出后是否执行了 onnx.checker.check_model() 和数值比对。

6.3 推理代码（inference.py / app 端）

  • 预处理对齐：确认 resize、normalize、permute 顺序与训练时完全一致。90% 的部署精度问题来自预处理不匹配。
  • 输出解析：检查 NMS 阈值、置信度阈值、top-k 等后处理参数是否与训练评估时一致。
  • 内存泄漏：端侧代码中是否及时释放 interpreter / session 资源。
  • 线程安全：TensorFlow Interpreter 和 ONNX Session 是否被多线程共享（通常不安全）。
preprocess_alignment_check.py — 预处理对齐验证
# 这是部署调试中最有价值的代码片段
# 同时运行 PyTorch 和 TFLite 预处理，对比像素级差异

def compare_preprocessing(image_path):
    # PyTorch 路径（训练时）
    img_pt = Image.open(image_path).convert('RGB')
    img_pt = img_pt.resize((224, 224))
    tensor_pt = transforms.ToTensor()(img_pt)           # [0,1]
    tensor_pt = transforms.Normalize(
        mean=[0.485, 0.456, 0.406],
        std=[0.229, 0.224, 0.225]
    )(tensor_pt)

    # TFLite 路径（Android/iOS 端常见）
    img_tf = Image.open(image_path).convert('RGB')
    img_tf = img_tf.resize((224, 224))
    arr_tf = np.array(img_tf).astype(np.float32) / 255.0  # [0,1]
    arr_tf = (arr_tf - [0.485, 0.456, 0.406]) / [0.229, 0.224, 0.225]
    arr_tf = np.expand_dims(arr_tf, axis=0)           # NHWC
    # TFLite 通常需要 NHWC，而 PyTorch 是 NCHW！
    arr_tf = np.transpose(arr_tf, (0, 3, 1, 2))         # 转为 NCHW 对比

    # 对比（差异应 < 1e-5）
    diff = np.abs(tensor_pt.numpy() - arr_tf)
    print(f"Max diff: {diff.max():.8f}")
    print(f"Mean diff: {diff.mean():.8f}")
    assert diff.max() < 1e-5, "预处理不一致！"

# 1. 加载模型权重
model = PiDiNetTiny()
checkpoint = torch.load("pidi_tiny.pth", map_location="cpu")
model.load_state_dict(checkpoint['state_dict'])
model.eval()

# 2. 调整 forward：确保输出为单张量
# 原始模型可能输出 list 或 dict，需包装
class WrappedPiDiNet(torch.nn.Module):
    def __init__(self, model):
        super().__init__()
        self.model = model

    def forward(self, x):
        outputs = self.model(x)
        # 如果输出是 list，取最后一个或拼接
        if isinstance(outputs, (list, tuple)):
            return outputs[-1]
        return outputs

wrapped = WrappedPiDiNet(model)

# 3. 导出 ONNX
dummy = torch.randn(1, 3, 512, 512)
torch.onnx.export(wrapped, dummy, "pidi_tiny.onnx",
                  opset_version=14,
                  input_names=["input"],
                  output_names=["edge_map"])

8.1 部署路径决策树

根据目标平台和模型类型，选择最优部署路径：

目标平台 视觉模型 (CNN) 语言模型 (LLM) 多模态
NVIDIA GPU 服务器 ONNX → TensorRT TensorRT-LLM / vLLM TensorRT
通用 CPU 服务器 ONNX Runtime llama.cpp (GGUF) ONNX Runtime
Android TFLite + GPU Delegate MLC-LLM / llama.cpp Android TFLite
iOS Core ML / TFLite llama.cpp iOS / MLX (Apple Silicon) Core ML
浏览器 ONNX Runtime Web WebLLM (WebGPU) ONNX Runtime Web
MCU / 嵌入式 TFLite Micro 不适用（模型过大） 不适用

8.2 核心经验法则

✓ 总是验证数值一致性

每次转换后，用同一批输入对比原始模型和转换后模型的输出。设置 rtol=1e-5 的严格阈值。数值不一致是部署 Bug 的首要来源。

✓ 预处理是最大陷阱

训练端的预处理（归一化、resize、channel 顺序）必须与推理端逐像素对齐。建议将预处理内嵌到模型图中（ONNX/TFLite）。

✓ 动态批大小是性能杀手

端侧模型固定 batch=1。服务端 LLM 推理使用 continuous batching（vLLM）或 inflight batching（TensorRT-LLM），而非静态批处理。

✓ 量化不是万能药

PTQ 对 CNN 有效，对 Transformer 需谨慎，对 LLM 必须使用专用量化方案（GPTQ/AWQ/GGUF）。量化后的模型必须重新跑评估指标。

✓ 算子覆盖检查先行

在模型设计阶段就查询目标推理框架的算子支持列表。一个不受支持的算子可能导致整图无法执行或大幅降速。

✓ 版本锁定工具链

ONNX opset、TensorFlow 版本、CUDA 版本之间的兼容性极其脆弱。将转换环境的版本写入 requirements.txt，并在 CI 中复现。

8.3 推荐工具链组合

环节 推荐工具 版本建议

最后一条建议：部署工作不是一次性任务，而是一个持续优化的过程。建立从训练到部署的自动化流水线（MLOps），每次模型迭代自动触发转换、验证、基准测试和回归测试，是规模化部署的唯一可行路径。
