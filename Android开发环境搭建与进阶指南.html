🖥️ 前置依赖详解

1. JDK（Java Development Kit）

背景：Android 应用本质上运行在 Java 虚拟机（ART）上，JDK 是编译 Java/Kotlin 代码的基础工具链。Google 官方推荐使用 JetBrains 维护的 OpenJDK，而非 Oracle JDK（授权问题）。

💡 为什么选 OpenJDK 17/21？

Android Studio 从 Arctic Fox 开始内置 OpenJDK，但独立安装系统级 JDK 能让你在命令行中使用 Gradle、keytool 等工具，且避免多个项目间 JDK 版本冲突。

安装步骤

PowerShell
# 推荐：使用 JetBrains Runtime 或 Eclipse Temurin
# 方式一：手动下载安装
# 访问 https://adoptium.net/ 下载 Eclipse Temurin JDK 17 MSI 安装包

# 方式二：使用 winget（Windows 包管理器）
winget install EclipseAdoptium.Temurin.17.JDK

# 验证安装
java -version
javac -version

环境变量配置

系统环境变量
JAVA_HOME = C:\Program Files\Eclipse Adoptium\jdk-17.0.x.x-hotspot
Path 追加：%JAVA_HOME%\bin

2. Git 版本控制

背景：Git 由 Linus Torvalds 于 2005 年创建，用于管理 Linux 内核源码。今天它已成为软件开发的事实标准，Android 开源项目（AOSP）本身就托管在 Git 上。

⚠️ Windows 下 Git 的换行符陷阱

Windows 使用 CRLF (\r\n)，而 Linux/macOS 使用 LF (\n)。Android 项目统一要求 LF，否则构建脚本可能报错。安装时选择 "Checkout as-is, commit Unix-style line endings"。

PowerShell
# 安装 Git
winget install Git.Git

# 配置全局设置
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
git config --global core.autocrlf input

# 验证
git --version

3. Windows Terminal（强烈推荐）

背景：Windows 传统的 cmd.exe 和 PowerShell 5 功能有限。Windows Terminal 是微软 2019 年开源的现代化终端，支持多标签页、GPU 加速、自定义主题，并能同时运行 PowerShell、WSL、CMD 等 Shell。

PowerShell
# 安装 Windows Terminal
winget install Microsoft.WindowsTerminal

# 推荐同时安装 PowerShell 7（跨平台、性能更好）
winget install Microsoft.PowerShell

🚀 Android Studio 安装与配置

背景：Android Studio 基于 IntelliJ IDEA Community Edition，由 Google 与 JetBrains 联合开发，2013 年发布首个预览版，2014 年成为官方 IDE。它集成了代码编辑、调试、性能分析、布局设计、APK 分析等全套工具，是 Android 开发的核心阵地。

安装步骤

1

下载安装包

访问 developer.android.com/studio 下载 Windows (64-bit) 版本。推荐选择 .exe 安装包（带自动更新）。

2

运行安装向导

勾选 Android Virtual Device（模拟器组件），安装路径建议改为非系统盘，如 D:\Android\Android Studio，避免 C 盘空间不足。

3

首次启动配置

选择 Do not import settings → 选择主题（Darcula 或 Light）→ 安装类型选 Standard → 确认 SDK 安装路径。

关键配置优化

1. 增大 IDE 内存

Android Studio 默认内存较低，大型项目容易卡顿。修改 studio64.exe.vmoptions：

Help → Edit Custom VM Options
-Xms1024m
-Xmx8192m
-XX:ReservedCodeCacheSize=512m
-XX:+UseG1GC
-XX:SoftRefLRUPolicyMSPerMB=50
-XX:CICompilerCount=2
-XX:+HeapDumpOnOutOfMemoryError
-Dfile.encoding=UTF-8

📱 模拟器与真机调试

Android Emulator 加速

背景：Android 模拟器历史上因性能差被诟病。2018 年 Google 推出基于 Hyper-V / HAXM 的硬件加速方案，配合 x86 系统镜像，性能已接近真机。

Windows 下启用硬件加速

PowerShell（管理员）
# 方案一：Hyper-V（Windows 10/11 专业版推荐）
dism.exe /Online /Enable-Feature:Microsoft-Hyper-V /All

# 方案二：Windows Hypervisor Platform（与 Hyper-V 二选一）
dism.exe /Online /Enable-Feature:HypervisorPlatform /All

# 方案三：Intel HAXM（仅 Intel CPU，旧方案）
# 在 SDK Manager 中下载 Intel x86 Emulator Accelerator

创建高性能模拟器

  1. 打开 AVD Manager（Tools → Device Manager）
  2. 选择设备定义（推荐 Pixel 7 或 Pixel 8）
  3. 选择系统镜像时，务必选择带 (Google Play) 或 (Google APIs) 的 x86_64 镜像
  4. 在高级设置中：
    • RAM：不少于 2048 MB（推荐 4096 MB）
    • VM heap：512 MB
    • Graphics：Hardware - GLES 2.0 或 3.0
    • Multi-core CPU：2 或 4 核

🛠️ 核心工具详解

以下是 Android 开发中不可或缺的工具，每个都说明了产生的背景和解决的问题。

平台工具

🐛ADB（Android Debug Bridge）

背景 | 通用调试桥梁

ADB 诞生于 Android 早期，是连接 PC 与 Android 设备的通用命令行工具。它解决的是"如何在没有图形界面的情况下与设备通信"的问题，是 Android 调试的基石。

核心用途：安装 APK、查看日志、端口转发、进入 shell、截图录屏、性能数据采集。

📊Android Profiler

背景 | Android Studio 3.0 集成

早期开发者需要用 DDMS、Traceview、MAT 等分散工具分析性能。Profiler 将它们统一为一个实时监控面板，解决"性能分析工具碎片化、学习成本高"的问题。

核心用途：CPU、Memory、Network、Energy 实时监控与方法追踪。

🎨Layout Inspector

背景 | 布局层级可视化

Android UI 是树形结构嵌套，过深的层级导致过度绘制和卡顿。Layout Inspector 让你实时查看每个 View 的属性、层级和渲染时间，解决"黑盒式 UI 调试"的痛点。

核心用途：查看视图层级、检查属性值、分析过度绘制。

📦APK Analyzer

背景 | APK 体积爆炸时代

随着依赖库增多，APK 体积动辄上百 MB。APK Analyzer 直观展示 APK 内部结构（DEX、资源、SO 库占比），解决"不知道体积膨胀根因"的问题。

核心用途：分析 APK 组成、比较版本差异、查看 DEX 方法数。

第三方调试神器

💧LeakCanary

Square 开源 | 内存泄漏检测

内存泄漏是 Android 最常见也最隐蔽的 Bug。LeakCanary 由 Square 公司开发，它通过监测 Activity/Fragment 的引用链，在泄漏发生时自动 dump 堆栈并生成可视化引用链，让"无法感知的内存泄漏"变得一目了然。

解决的问题：Activity 因匿名内部类、静态引用、未注销监听器等原因无法被 GC 回收。

🔄Flipper（原 Sonar）

Meta(Facebook) 开源 | 移动端调试平台

Flipper 是 Facebook 推出的移动端调试平台，它将移动端内部状态以桌面应用的形式可视化展示。解决"移动端调试信息封闭、难以实时查看"的问题。

核心插件：网络请求拦截、数据库浏览、SharedPreferences 查看、布局检查、崩溃日志。

🌐Charles / Fiddler

网络抓包工具 | HTTPS 解密

移动应用的网络请求发生在设备与服务器之间，传统 Wireshark 学习成本高。Charles 以中间人代理的方式拦截 HTTP/HTTPS 流量，解决"移动端网络请求黑盒"问题。

核心用途：查看请求/响应、Mock 接口数据、弱网模拟、HTTPS 解密。

🔍Stetho

Meta(Facebook) 开源 | Chrome DevTools 集成

让 Android 应用能够通过 Chrome 开发者工具进行调试。它解决的是"Android 开发者习惯 Web 调试工具但移动平台没有等价物"的问题。

核心用途：数据库查看、网络监控、View 层级检查（部分被 Flipper 取代）。

🧪Android Lint

Google 官方 | 静态代码分析

Lint 是 Android 官方的静态分析工具，它在编译前扫描代码中的潜在问题。解决"代码质量问题发现滞后、代码规范难以统一"的问题。

检测范围：性能、可用性、版本兼容性、安全性、国际化、正确性。

📏Systrace / Perfetto

Google 官方 | 系统级性能追踪

Systrace 是 Android 系统级的性能追踪工具，能采集 CPU 调度、渲染管线、Binder 通信等底层数据。解决"应用卡顿但不知道是 App 问题还是系统问题"的困境。

Perfetto 是 Systrace 的下一代，支持更大数据量和 SQL 查询。

ADB 高级命令速查

ADB 常用命令
# 设备管理
adb devices                    # 列出已连接设备
adb connect :5555         # 无线调试（需先 USB 连接执行 adb tcpip 5555）
adb disconnect                # 断开无线连接

# 应用管理
adb install app.apk           # 安装 APK
adb install -r app.apk        # 覆盖安装（保留数据）
adb uninstall com.example.app # 卸载应用
adb shell pm clear com.example.app  # 清除应用数据

# 日志
adb logcat                    # 查看全部日志
adb logcat -s TAG:D           # 过滤 TAG，级别 Debug 以上
adb logcat -c                 # 清除日志缓冲区
adb logcat | findstr "keyword" # Windows 下过滤（Linux/macOS 用 grep）

# 文件传输
adb push local.txt /sdcard/   # 推送到设备
adb pull /sdcard/file.txt ./  # 拉取到电脑

# Shell 操作
adb shell                     # 进入设备 shell
adb shell am start -n com.example/.MainActivity  # 启动 Activity
adb shell input keyevent 26   # 模拟电源键
adb shell screencap /sdcard/screen.png  # 截图
adb shell screenrecord /sdcard/video.mp4 # 录屏（Ctrl+C 停止）

# 性能
adb shell dumpsys meminfo com.example.app  # 内存详情
adb shell dumpsys battery     # 电池状态
adb shell top -m 10           # 查看 CPU 占用 top10

🐞 调试技巧大全

1. 断点高级玩法

断点类型 用法 适用场景
条件断点 右键断点 → 输入条件表达式，如 userId == 12345 循环中只在特定值暂停
日志断点 勾选 "Evaluate and log"，不暂停只输出 不打断程序流，只观察变量值
异常断点 Run → View Breakpoints → 添加 Java Exception Breakpoints 捕获特定异常抛出的位置
方法断点 在方法签名上打断点 跟踪方法出入参（性能开销大，慎用）
Field Watchpoint 在字段上打断点，监控读写 追踪某个变量何时被修改

2. 卡顿定位三板斧

第一板斧：GPU 过度绘制检测

开发者选项
设置 → 开发者选项 → 调试 GPU 过度绘制 → 显示过度绘制区域

颜色含义：
蓝色   = 绘制 1 次（理想）
绿色   = 绘制 2 次（可接受）
粉色   = 绘制 3 次（需优化）
红色   = 绘制 4 次以上（必须优化）

第二板斧：Systrace 分析渲染管线

命令行

python systrace.py -a com.example.app sched gfx view wm am -b 32768 -t 5 -o trace.html

# 在 Chrome 中打开 trace.html
# 关注 "Frame" 行，绿色表示 16ms 内完成，黄色/红色表示掉帧

第三板斧：Profiler CPU 方法追踪

  1. 打开 Profiler → 选择 CPU 标签
  2. 选择 "Trace Java Methods"
  3. 录制卡顿发生时段
  4. 查看 "Bottom Up" 视图，找出耗时最长的方法调用链

3. 内存泄漏排查流程

1

集成 LeakCanary

在 build.gradle 添加依赖，它会自动监控 Activity 和 Fragment。

2

复现场景

反复进入/退出可疑页面，触发 LeakCanary 的泄漏检测。

3

分析引用链

LeakCanary 会生成通知，展示从 GC Root 到泄漏对象的完整引用链。

4

确认根因

常见根因：静态引用持有 Activity、匿名内部类持有外部类引用、未注销监听器、Handler 延迟消息。

4. 多模块项目设计原则

  • 按功能拆分：feature-login、feature-home、feature-search，而非按层拆分（data、domain、ui）
  • 依赖方向严格单向：App 模块 → Feature 模块 → Core 模块，禁止反向依赖
  • API 显式暴露：每个模块通过 api/implementation 精确控制对外暴露的内容
  • 版本集中管理：在根目录 build.gradle.kts 或 libs.versions.toml 统一管理依赖版本

🚀 性能优化体系

启动优化

应用启动是用户体验的第一道关卡。Android Vitals 监控冷启动时间，超过 5 秒会影响 Google Play 排名。

Application.onCreate 优化
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        
        // ❌ 错误：所有初始化放在主线程
        // initNetwork()
        // initDatabase()
        // initPush()
        // initAnalytics()
        
        // ✅ 正确：按优先级和线程分流
        initCriticalOnMainThread()     // 必须在主线程且立即执行
        
        GlobalScope.launch(Dispatchers.IO) {
            initDatabase()             // 数据库可以异步
            initNetwork()
        }
        
        mainExecutor.executeDelayed(2000) {
            initPush()                 // 推送可以延迟
            initAnalytics()
        }
    }
}

内存优化 checklist

  • 使用 LeakCanary 定期检查内存泄漏
  • Bitmap 加载使用 Glide/Coil，不要手动 BitmapFactory.decode
  • List 使用 RecyclerView 而非 ListView，做好 ViewHolder 复用
  • 避免在 onDraw 中创建对象（引起 GC）
  • 使用 SparseArray 替代 HashMap<Integer, Object>
  • 谨慎使用枚举（每个枚举值都是一个对象）
  • Fragment/Activity 中使用弱引用持有外部对象

🔋 Doze 模式与后台限制

Android 6.0+ 引入 Doze 模式，系统会限制后台网络访问和 CPU 唤醒。开发时需使用 WorkManager 替代后台 Service，使用 FCM 替代轮询。

WorkManager 后台任务
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED)
    .setRequiresBatteryNotLow(true)
    .build()

val uploadWork = PeriodicWorkRequestBuilder<UploadWorker>(1, TimeUnit.HOURS)
    .setConstraints(constraints)
    .build()

WorkManager.getInstance(context).enqueueUniquePeriodicWork(
    "upload",
    ExistingPeriodicWorkPolicy.KEEP,
    uploadWork
)

典型项目结构
app/
├── data/                      # 数据层

│   │   ├── HomeViewModel.kt   # 状态持有者
│   │   └── HomeUiState.kt     # UI 状态封装
│   └── common/                # 通用组件
└── di/                        # 依赖注入（Hilt 模块）

关键架构组件

📌ViewModel

AAC 组件 | 配置变更存活

屏幕旋转等配置变更会导致 Activity 重建，ViewModel 在作用域内保持存活，解决"配置变更导致状态丢失"的问题。它是 UI 数据的持有者，不持有 View 引用。

🔄Repository 模式

数据来源抽象层

ViewModel 不直接调用网络或数据库，而是通过 Repository。Repository 决定数据从哪来（本地缓存 or 远程服务器），解决"ViewModel 耦合具体数据来源"的问题。

💉Hilt（依赖注入）

Dagger 上层封装 | 编译期生成

手动创建对象导致耦合严重、难以测试。Hilt 在编译期自动生成工厂代码，零运行时反射开销，解决"依赖管理混乱、单元测试难以 mock"的问题。

🌊Kotlin Coroutines + Flow

异步编程模型

替代 RxJava 和 AsyncTask，编译器支持、语法简洁、结构化并发自动处理生命周期。Flow 是响应式数据流，完美适配 LiveData 到 StateFlow 的演进。

Compose 声明式 UI

背景：传统的 View 体系基于命令式编程（findViewById → 修改属性），代码冗长且容易不一致。Jetpack Compose 是 Google 2021 年推出的全新声明式 UI 框架，用 Kotlin 函数描述 UI，框架自动处理刷新。

Compose 示例
@Composable
fun UserProfile(user: User) {
    Column(
        modifier = Modifier
            .fillMaxWidth()
            .padding(16.dp)
    ) {
        AsyncImage(
            model = user.avatarUrl,
            contentDescription = "头像",
            modifier = Modifier
                .size(80.dp)
                .clip(CircleShape)
        )
        Spacer(modifier = Modifier.height(8.dp))
        Text(
            text = user.name,
            style = MaterialTheme.typography.headlineSmall
        )
        Text(
            text = user.bio,
            style = MaterialTheme.typography.bodyMedium,
            color = MaterialTheme.colorScheme.onSurfaceVariant
        )
    }
}
💡 Compose 学习建议
  • 理解 重组（Recomposition） 机制：什么情况下会触发、如何避免不必要重组
  • 掌握 remember / rememberSaveable 状态管理
  • 使用 derivedStateOf 避免频繁计算
  • 利用 LaunchedEffect / SideEffect / DisposableEffect 处理副作用
